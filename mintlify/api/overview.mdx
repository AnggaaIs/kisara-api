## Authentication

Protected endpoints for API consumers use API key authentication.

<CodeGroup>

```http HTTP Header
X-API-Key: <your_api_key>
```

```bash cURL Example
curl -X GET "https://api.kisara.my.id/endpoint" \
  -H "X-API-Key: <your_api_key>"
```

</CodeGroup>

<Warning>
  Requests without a valid API key will be rejected. Store API keys securely and avoid exposing them in public clients.
</Warning>

<Info>
  JWT bearer token is still used for internal web session flows, such as login/refresh and API key lifecycle management.
</Info>

## Rate Limiting

The API applies both global and per-route rate limits to keep the service stable.

Current limits:

- Global (all routes): 100 requests per IP per minute
- Auth and API key management routes (`/auth/*`): 10 requests per IP per minute
- User profile route (`/user`): 5 requests per IP per minute
- Message routes (`/message/*`): 30 requests per IP per minute

Notes:

- If a route has a stricter route limit, that stricter limit is applied first.
- Home routes use the global limit.
- Rate limit key is based on client IP.

When a limit is exceeded, the API responds with HTTP `429` and the standard error envelope.

```json
{
  "status_code": 429,
  "message": "Rate limit exceeded",
  "metadata": {
    "method": "GET",
    "url": "/endpoint",
    "execution_time_ms": 1,
    "ip": "127.0.0.1",
    "timestamp": "2026-04-15T12:00:00.000Z"
  }
}
```

<Steps>
  <Step title="Detect temporary failures">
    Handle HTTP `429` as a temporary failure.
  </Step>
  <Step title="Retry with backoff">
    Retry with exponential backoff (for example: 1s, 2s, 4s, max 30s).
  </Step>
  <Step title="Reduce burst traffic">
    Use batching or queues for heavy endpoints and parallel operations.
  </Step>
</Steps>

## Response Format

Most endpoints return a response envelope like this:

```json
{
  "status_code": 200,
  "message": "OK",
  "data": {},
  "metadata": {
    "method": "GET",
    "url": "/endpoint",
    "execution_time_ms": 12,
    "ip": "127.0.0.1",
    "timestamp": "2026-04-14T12:00:00.000Z"
  }
}
```

<Info>
  Treat this response shape as your main contract. Always validate `status_code` and metadata fields on the client side.
</Info>

## Best Practices

<CardGroup cols={2}>
  <Card title="Always check status_code" icon="circle-check">
    Don't rely on HTTP status alone. Use `status_code` and `message` to validate outcomes.
  </Card>
  <Card title="Keep metadata" icon="database">
    Log `metadata.timestamp` and `metadata.execution_time_ms` for audits and debugging.
  </Card>
  <Card title="Use timeouts" icon="timer">
    Configure client-side request timeouts so your app doesn't hang during network issues.
  </Card>
  <Card title="Use idempotency" icon="repeat">
    For sensitive operations, prevent duplicate requests to avoid side effects.
  </Card>
</CardGroup>
